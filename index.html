MarketPlace上で動作する検索エンジン及びDBの開発

Feb 2020 – Present public

要件

  • MarketPlace上で参照するための顧客データを持つDBを用意する
  • MarketPlace上で動作する自然言語処理ロジックを構築する
  • 顧客データを顧客自らuploadするための管理画面を用意する
  • 管理画面からDBへデータをpushするためのpipelineを用意する
  • 各サービスはDockerコンテナで管理し、Kubernetesで管理する。

前提

  • MarketPlaceの検索エンジンであるため、高トラフィックに耐えうる設計であるべき
  • MarketPlace内の顧客情報には、商品の在庫状況が含まれており、更新処理が頻繁に発生することが予想される。

担当工程

  • 自然言語処理ロジックの要件定義・実装まで
  • 顧客データを持つDBのindex設計・実装
  • データパイプラインの設計・実装
  • (管理画面の設計・実装)

採用した技術

ElasticSearch

採用理由

  • Lucene をベースとした ElasticSearch か Solr を採用対象とし、結果として ElasticSearch を採用。
  1. 柔軟に検索ロジック  の構築をしたいので、plugin が豊富で簡単に拡張できる ElasticSearch が良さそうだった。Solr の場合は Jar ファイルを都度用意する必要があり、チームの規模的に拡張性とコストが見合わなかった。
  2. BI ツール及び、GUI として kibana を採用したかった。振り返ってみるとデータの投入や、クエリ作成など kibana を導入することでとても捗った。solr にも banana という BI ツールがあったのだが、個人的に kibana のほうが UI は素敵だった。
  3. 同様の MarketPlace 構築に向け GMO が ElasticSearch を採用、また、マッチングアプリや、グルメサイト等も ElasticSearch を採用しており、技術的に参照できそうなドキュメントが多かった。

    https://www.slideshare.net/ssuser7eadbf/elasticsearch-sudachi

  4. 他の project で、インシデント時の alert から、データの push などで Fluentd を採用している。今回も、データパイプラインの中で fluentd を採用したいため、Fluentd に plugin を持つ ElasticSearch を採用したい。 　

k8s

採用理由

  • 顧客データfileを管理画面からアップロードした際、ElasticSearch投入する前に、下記2点の処理が必要になる。

    1. Fileをinsertできるデータ形式に変換
    2. 変換したデータをElasticSearchに投入

  • 顧客データは数万行に及ぶため、Clientで処理することは避けたい。

  • 上記2つの責務を果たすInterfaceをElasticSearchとClient間に用意する必要がある。

  • この際、2つの責務を1つのInterfaceでは果たそうとした場合、下記が心配。

    1. ElasticSearchへデータuploadに失敗した際、リトライ用にデータはInterface上に持つ必要がある。
    2. その場合、データが増加するに従って、Interfaceのパフォーマンスが落ちそう。
    3. 今後スケールする際に、責務別にインスタンスサイズを調整できない。

  • そのため、下記責務別にApplicationを用意し、Interfaceのみに責務を持たないようなデータパイプラインを構築したい。

    • 管理画面からアップロードされたファイルを、データに変換するInterface
    • リトライ用にデータを保存するBucket
    • Bucketを参照し、ElasticSearchにデータを分散するフレームワーク

  • 上記のデータパイプラインを構成するContainerのサービスイン・運用コストを抑えるために、k8sを採用したい。

Fluentd

採用理由

  • 分散処理用のフレームワーク。ElasticSearch用のPluginがあったので採用。
  • 前回のPJで触ったとき、監視機能が有用だったこともある。

Kafka

採用理由

  • 今回は顧客データ内に商品在庫を持つ。更新が発生する際、逐次処理したいので、メッセージキューを採用したかった。
  • また、商戦の時期に入ると、データ量が急増し、DBを逼迫することが予想されるので、負荷を均したかった。 -上記のデータパイプラインにKafkaを追加。

要件

  • 高頻度のトラフィックに耐えうる設計であること。
  • KVS形式でデータを管理すること。

前提

  • 既存のサービスに対する機能追加

担当工程

  • 要件定義〜実装まで

採用した技術

Docker

AWS DynamoDB

採用した理由

  • 今回は、KVS形式でデータを管理するかつ、トランザクションが想定されないため、NoSQLを採用したい。

  • また、リクエストが膨大であり、今後スケールの可能性が高い。

  • NoSQLの中でも、オートスケールを強みとするDynamoDBを採用。レプリケーションやfailoverを自動で再現してくれる点も、運用コストを下げるという意味で評価した。

AWS EC2

採用した理由

  • 大量のリクエストが予想されるため、リクエスト単位で料金が発生するAWS Lambdaは避けたい。
  • リクエストの規模が想定できないため、リリース後しばらくはDynamoDBのキャパシティーモードに オンデマンドを採用したい。 => AWS Elastic Beanstalkは採用しない => EC2上にアプリケーションを構築

Node.js

採用理由

  • DynamoDBを採用するため、aws-sdkをサポートしている必要があった。
  • 使用上発行したキーが重複する可能性を避けたいのでスレッドセーフなApplicationを構築したかった。
  • 今後自社内の他のサービスでも用いられる可能性が高かったので、可読性が高いかつ、学習コストの低い言語で再現したかった。

webpack

採用理由

  • アプリケーションのbuildをタスクランナーで実行したい。
  • 技術的なチャレンジという側面も強い。

Terraform

採用理由

  • 今後サービスごとに当Applicationを拡張機能として追加できるようにしたい。
  • DynamoDBにてTTLの設定や、Primary Keyの設定、特に、キャパシティモードについての設定はコード管理しておきたい。

オフラインイベント用のモバイルアプリケーション開発

Jul 2019 – Oct 2019

要件

  • オフラインイベントの進行をモバイルアプリで行いたい
  • AndroidとiPhoneの両方でリリースする
  • 会場Map上にイベント開催位置が表示される
  • イベントの開催時間に通知を送る
  • ユーザーがいいねしたイベントをMap上で強調する・時間になったら開始通知をする
  • 入場用チケットをアプリケーション上から確認する
  • 運営の任意のタイミングで通知を送信できるようにする
  • etc

前提

  • 割り当てられる人員を考慮し、マルチプラットフォームを採用したい

担当工程

  • 要求分析〜実装まで
  • リリース作業についてはiOSのみを担当

採用した技術

Flutter

採用理由

  • 選択肢として、React NativeかFlutter、Xamarinがあったが、React Nativeは3rd Partyのライブラリが豊富である反面、障害点の切り分けが困難であることから採用を避けた。、Xamarinについてはネット上のドキュメントがやや薄い用に感じたため採用を避けた。

  • Flutterを選択したのは、JavaScriptライクなDartで書くことができたこと、公式のライブラリ・ドキュメントが豊富であったため。

  • 終了後に思ったことだが、まだまだ歴が浅いフレームワークであり、公式のpluginでも予期しない挙動が発生したり、各Widget間でのstateの共有パターンが複数勃興しているかつ難解なものもある。一時期のReactのstate論争に似たものを感じた。

Firebase

採用理由

  • プッシュ通知を実装する際、公式のfirebasePluginが用意されていたため。
  • flutter側のpluginのversionと、Flutter自体のversionに互換性がなく、挙動を確認する際はFlutterのversionをダウングレードするなどしなければ行けなかった。今後改善されるとは思うが、コミュニティがまだまだ途上である気がする。貢献していきたい。

Cybozu Garoon APIを用いたSlackbotの開発

Sep 2018 – Oct 2018

要件

  • 導入しているグループウェアの管理画面を経由せずにSlackから社内情報にアクセスする
  • 社内情報にアクセスするためのキーは外部に公開しない

前提

  • 社内ではSlackが展開されている
  • 社内インフラはAWS
  • グループウェアとしてCybozuを採用

担当工程

  • 要件定義〜実装までを担当。

採用した技術

AWS Lambda ・ API Gateway ・ Cloudwatch Logs

採用理由

  • 社内ツールのため大量のリクエストは想定されず、実行が不定期になりそうだった。

  • APIを叩いた結果を返すだけなので、リクエストに対して負荷の高い処理は発生せず、実行時間も短そう。

    => 比較的サービスがコールドな状態にあることが想定されるため、EC2インスタンスを採用するよりも、リクエスト単位で課金のAWS Lambdaを採用したほうがコスパが良さそうだった。

    => 処理の時間も短そうなので、処理の負荷も課金単位であるLambdaでも問題なさそうだった。

  • 運用・管理コストをかけたくなかった。

  • 選択肢としてGoogle Apps Scriptもあったが、社内情報にアクセスするためのキーを、私用のGoogleアカウントで管理するのは避けたかったため却下。

  • Gsuiteアカウント権限を変更すれば可能ではあったが、運用チームの手間と、ツールが及ぼすインパクトを比較した際に、釣り合わなかったのでそれも却下。

Node.js

採用理由

  • APIのレスポンスがSOAP形式であるため、対象となる情報を取り出すため、柔軟に文字列操作したかった。（SOAP API用のクライアントライブラリを導入すると、Lambdaのコストが高くなりそうだった） 　

  • 上記含め、技術的な縛りはあまりなかったため、自社サービスに親和性の高いNode.jsを採用した。

  • 初学者のスタートアップとして触ってもらうプロジェクトにしようという裏テーマがあったものの、非同期処理の性質を持つNode.jsはやや初学者にとってはハードルが高い気がする。
