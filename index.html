広告の検索精度を改善するプロジェクト

Position: Developer

Feb 2020 – Jul 2020 public

前提

同上

背景

  1. 商材に関するデータは、OracleDB内に格納されている。
  2. ブロガーが商材を検索する際は、検索ワードを元に、LIKE検索を実行するクエリをOracleDBに対して実行し、検索結果を表示している。

理想

現行のLIKE検索より、検索精度の高い検索を実現したい

手法

ElasticSearchを導入し、OracleDB内に格納されているデータをElasticSearchに同期させ、検索クエリの実行先をElasticSearchに切り替える。(下図には、後述するプロジェクトに関するComponentも記載されている)

担当工程

  1. 技術選定(所属チームで議論)
  2. 形態素解析のロジック考案・実装
  3. ElasticSearchのテーブル設計・実装

採用した技術

ElasticSearch

検索精度の向上を目的として採用されていることが多かった、ElasticSearch(以降ESと呼ぶ) か、ESと比較されやすいSolr　を比較し、結果 ES を採用した。

［採用理由］

  1. 柔軟に検索ロジック  の構築をしたいので、plugin が豊富で簡単に拡張できる ES が良さそうだった。一応調査したが、Solr の場合は、Pluginを用意するために、 Jar ファイルを都度用意する必要があり、めんどくさそうだった。

  2. BI ツール及び、GUI として kibanaがあるのがかなり、魅力的だった。振り返ってみるとテストデータの投入や、クエリ作成など kibana によってとても捗った。solr にも banana という BI ツールがあるらしいが、個人的に kibana のほうが UI は素敵だった。

  3. 同様の MarketPlace 構築に向け GMO が ES を採用、また、マッチングアプリや、グルメサイト等も ES を採用しており、技術的に参照できそうなドキュメントが多かった。

    https://www.slideshare.net/ssuser7eadbf/elasticsearch-sudachi

成果

OracleDBからESにデータを同期するために、ESで公式に推奨されているlogstashを利用することを検討している。

プロジェクトに着手していた当時、別チームにてOracleDBの移行プロジェクトが同時進行していたため、データの同期自体は次回のマイルストーンとして設定し、データを受け入れるESのみを構築し、プロジェクトを完了した。データの同期を完了することで、検索精度の改善が見込まれる。

https://www.elastic.co/jp/blog/how-to-keep-elasticsearch-synchronized-with-a-relational-database-using-logstash

暗号化APIを開発するプロジェクト

Position: Developper

Nov 2019 – Jan 2020 public

前提

同上

背景

アフィリエイトサービスでは、URL内にクエリパラメータを付与することで、個人を識別する仕組みを実現している。

ex) https://affiliate/index.html?user_name=田中太郎

アフィリエイトサービスを複数利用している会社は、このURL内に、複数社分のクエリパラメータが付与され、URLが長くなってしまう。

ex) https://affiliate/index.html?companyA_user_name=田中太郎&companyB_user_name=田中太郎&companyC_user_name=…

URLはBrowserごとに最大文字数が決まっており、それ以上の文字数になると、自動的にURLをカットされてしまう。

ex) https://affiliate/index.html?companyA_user_name=田中太郎&companyB_user_name=田中太郎&companyC_user_name=…companyZ_us(ここで終わり)

そのため、他社への影響を押さえるため、自社のクエリパラメータを、暗号化することとした。

- companyA_user_name=田中太郎(実際はもっと長い)
+ companyA_user_name=XSGW1GSB

手法

発行されたクエリパラメータを暗号化してURLに付与し、コンバージョン(アフィリエイトが成立)の際は、暗号化したキーをクエリパラメータに戻して、既存の成果発生処理プロセスに流す。

担当工程

要件定義〜実装まで

採用した技術

AWS DynamoDB

［採用理由］

  1. 暗号化の性質上、データ形式が1:1であったこと
  2. 複雑なトランザクションも想定されなかったこと

から、NoSQLを採用することにした。

AWSを触っていたこともあり、NoSQLの中でも、DynamoDBを採用した。オートスケールに加え、レプリケーションやfailoverを自動で再現してくれる点も、運用コストを下げられそうだったので、嬉しかった。

Node.js

［採用理由］

シンプルなアプリケーションであったため、aws-sdkがサポートしている言語(DynamoDBを採用するため)内で、お手軽なNode.jsを採用した。 　

成果

課題を解決済み。正常に動作している。

理想

オフラインイベントアプリの基本的な機能を実装したい。
