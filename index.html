アフィリエイトバナーを配信するDBの検索精度を改善するプロジェクト

Position: Developer

Feb 2020 – Present public

背景

  1. アフィリエイトバナーに関するデータは、OracleDB内に格納されている。
  2. アフィリエイトバナー検索画面から、ユーザーがアフィリエイトバナーを検索する際は、DBに対して、検索ワードでLIKE検索を実行するクエリを発行し、検索結果を返している。

理想

現行のLIKE検索より、検索精度の高い検索を実現したい

手法

ElasticSearchを導入し、OracleDBのデータとElasticSearchのデータを同期させ、検索ワードはElasticSearchに投げるようにする。

担当工程

  1. 技術選定(所属チームで議論)

  2. 形態素解析のロジック考案・実装

  3. ElasticSearchのテーブル設計・実装

採用した技術

ElasticSearch

検索精度の向上を目的として採用

検索精度の向上を目的として採用されていることが多かった、ElasticSearch(以降ESと呼ぶ) か、ESと比較されやすいSolr　を比較し、結果 ES を採用した。

【採用理由】

  1. 柔軟に検索ロジック  の構築をしたいので、plugin が豊富で簡単に拡張できる ElasticSearch が良さそうだった。Solr の場合は、Pluginを用意するために、 Jar ファイルを都度用意する必要があり、めんどくさそうだった。

  2. BI ツール及び、GUI として kibanaがあるのがかなり、魅力的だった。振り返ってみるとテストデータの投入や、クエリ作成など kibana によってとても捗った。solr にも banana という BI ツールがあるらしいが、個人的に kibana のほうが UI は素敵だった。

  3. 同様の MarketPlace 構築に向け GMO が ElasticSearch を採用、また、マッチングアプリや、グルメサイト等も ElasticSearch を採用しており、技術的に参照できそうなドキュメントが多かった。

    https://www.slideshare.net/ssuser7eadbf/elasticsearch-sudachi

Python

データをOracleDBからESに投入する際に採用

OracleDBから同期するテーブル以外にも、ES内に追加したいテーブルが出てきた。

新たにESに追加するテーブルには、様々なデータフォーマットのデータを投入されるため、ESに投入する前段階で、データフォーマットを統一して、ESに投入するためのAdapterを実装するために、Pythonを採用した。

【採用理由】

データフォーマットを整える際、Pandasを使いたかった。

成果

結論からいうと、当初想定していた検索パフォーマンスを達成できているとはまだ言えない。 その原因は、OracleDBからのデータはElasticSearchに同期しない方向で実装を進めたことにある。

理想だった手法

本来はOracleDBのデータをESに同期し、ESを参照したかったが、OracleDBからESにデータを同期するために検討した方法に、いづれも問題点があった。

方法1. Logstashを用いてESにデータを流す

ESで公式に推奨されている方法。

https://www.elastic.co/jp/blog/how-to-keep-elasticsearch-synchronized-with-a-relational-database-using-logstash

この方法で行きたかったが、OracleDBはオンプレで動作しており、Logstash導入時の影響範囲が想定できなかった。 近々Cloudへの移行の話もあるとのことだったので、一旦この方法は見送った。

方法2.定期的にOracleDBからデータを取得するbatchを用意する

OracleDBから指定テーブルのデータを落とすバッチを用意し、そのバッチサーバ内のデータをESにuploadする方法を考えた。

しかし、この方法だと、下記2点の嫌なことがあったので、この方法も見送った。

  1. 差分更新ではなく、都度全て更新してしまう
  2. データが大きいため、バッチの動作〜ESへのデータアップロードまでのフローで、OracleDB〜ES間のデータに、30分以上のラグが出てしまう

代替案

検討した結果、タイムラグを許容してまで、検索精度を上げる必要はないという結論になった。

なので、OracleDBのCloud化に伴い、Logstashを導入できるまでは、

  1. 検索画面から、OracleDBに存在しないテーブルのみを持つESに対して、検索ワードを投げる
  2. ESからのレスポンス内に存在する、共有のプライマリーキーで、OracleDBを参照する
  3. その結果をMergeして、検索画面に返す

という、クライアント側で擬似的なjoinを行う方法を採用することにした。

※ES内にのみ新規追加したテーブルと、OracleDB内のテーブルは、共通のプライマリーキーを持っている

ES内に新規追加される、OracleDBには存在しないテーブル

アフィリエイトID new_columnA …
001 … …
002 … …

ESに同期したかった、OracleDB内のテーブル

アフィリエイトID exist_columnB …
001 … …
002 … …

代替案の惜しい点

ESに設定した、N-gram・形態素解析が、既存のOracleDBのテーブルには適用されない点。 検索でHitしてほしい情報が、ES内に網羅的に存在する状態が理想なので、OracleDBがCloudに移行したタイミングで、OracleDB〜ES間のデータを同期できるようにしたい。

 
 
 
 
 

Project: 暗号化APIの開発

Position: Developper

Nov 2019 – Jan 2020 public

背景

アフィリエイトサービスでは、URL内にクエリパラメータを付与することで、個人を識別する仕組みを実現している。

ex) https://affiliate/index.html?user_name=田中太郎

アフィリエイトサービスを複数利用している会社は、このURL内に、複数社分のクエリパラメータが付与され、URLが長くなってしまう。

ex) https://affiliate/index.html?companyA_user_name=田中太郎&companyB_user_name=田中太郎&companyC_user_name=…

URLはBrowserごとに最大文字数が決まっており、それ以上の文字数になると、自動的にURLをカットされてしまう。

ex) https://affiliate/index.html?companyA_user_name=田中太郎&companyB_user_name=田中太郎&companyC_user_name=…companyZ_us(ここで終わり)

そのため、他社への影響を押さえるため、自社のクエリパラメータを、暗号化することとした。

- companyA_user_name=田中太郎(実際はもっと長い)
+ companyA_user_name=XSGW1GSB

手法

発行されたクエリパラメータを暗号化してURLに付与し、コンバージョン(アフィリエイトが成立)の際は、暗号化したキーをクエリパラメータに戻して、既存の成果発生処理プロセスに流す。

担当工程

要件定義〜実装まで

採用した技術

AWS DynamoDB

KVS形式のデータストアとして採用

［採用理由］

  1. 暗号化の性質上、データ形式が1:1であったこと
  2. 複雑なトランザクションも想定されなかったこと

から、NoSQLを採用することにした。

AWSを触っていたこともあり、NoSQLの中でも、DynamoDBを採用した。オートスケールに加え、レプリケーションやfailoverを自動で再現してくれる点も、運用コストを下げられそうだったので、嬉しかった。

Node.js

暗号化アプリケーションとして採用

［採用理由］

シンプルなアプリケーションであったため、aws-sdkがサポートしている言語(DynamoDBを採用するため)内で、お手軽なNode.jsを採用した。

 
 
 
 
 

Project: オフラインイベント用のモバイルアプリケーション開発

Position: Developper

Jul 2019 – Oct 2019

背景

オフラインイベント開催に向け、来場者向けにモバイルアプリケーションの開発依頼を受けた。

理想

オフラインイベントアプリの基本的な機能を実装したい。

  • イベント運営者が希望するタイミングでpush通知をアプリに送信する
  • オフラインイベント内の各企画に対して、いいね・Share機能を追加する等

担当工程

要求分析〜実装・リリース作業(リリース作業はiOSのみを担当)

採用した技術

Flutter

クロスプラットフォームフレームワークとして採用

［採用理由］

選択肢として、React NativeかFlutter、Xamarinがあったが、React Nativeは3rd Partyのライブラリが豊富である反面、障害点の切り分けが困難であることから採用を避けた。Xamarinについてはネット上のドキュメントがやや薄い用に感じたため採用を避けた。

Flutterを選択したのは、JavaScriptライクなDartで書くことができたこと、公式のライブラリ・ドキュメントが豊富であったため。

終了後に思ったことだが、まだまだ歴が浅いフレームワークであり、公式のpluginでも予期しない挙動が発生したり、各Widget間でのstateの共有パターンが複数勃興しているかつ難解なものもある。
